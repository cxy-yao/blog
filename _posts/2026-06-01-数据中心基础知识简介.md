## 前言

数据中心从传统数据机房发展到云数据中心，伴随着虚拟化、云计算、容器、人工智能等技术的部署使用。数据中心网络作为承载企业业务的基础，也在不断发展和演进。本文基于华为 HCIE-Data Center Network V1.0 培训课件，系统梳理数据中心的发展、IT技术和网络技术概况。

## 正文

### 1. 数据中心概述

#### 1.1 什么是数据中心

数据中心（Data Center）是一整套包括建筑在内的复杂设施。它不仅包括计算机系统和其它与之配套的系统（例如通信和存储系统），还包含数据通信连接、环境控制设备、监控设备以及各种安全装置。

数据中心有**分层架构**，从L0到L4可以分为五个层次：



| 层次 | 名称 | 说明 |
| :--: | :--: | :-- |
| L0 | 土建及配套 | 土建、园区、楼宇，市政电/水引入 |
| L1 | 基础设施层 | IT模块、电力能源、其他辅助系统，物理环境与硬件设备 |
| L2 | 硬件设备层 | 计算、存储、网络，ICT设备 |
| L3 | 平台系统层 | 软件、平台与应用 |
| L4 | 业务应用 | 智慧金融、智慧城市、公共安全、企业办公、电商平台等 |

#### 1.2 数据中心的分类



**按容量分类：**

| 类型 | 标准机架数 |
| :--: | :--: |
| 超大型数据中心 | ≥ 10,000 个 |
| 大型数据中心 | 3,000 ~ 10,000 个 |
| 中小型数据中心 | < 3,000 个 |

**按用户类型分类：**

- **EDC**（Enterprise Data Center，企业数据中心）：企业自用的数据中心，是企业的支撑系统。稳定、可靠是首要需求。
- **IDC**（Internet Data Center，互联网数据中心）：数据中心领域的"运营商"，对外提供整租或零售服务。拥有完善的功能（包括高速互联网接入带宽、高性能局域网络）、安全可靠的机房环境、专业化的管理和应用服务平台。

**按建设方式分类：**

- 传统定制建设模式
- 预制模块化模式

**按基础设施标准分类：**

- Uptime Institute（美国）Tier分级
- TIA-942 美国电信工业协会标准
- GB50174 国家标准

#### 1.3 数据中心的行业分布

数据中心分布千行百业，是数字化转型的基础设施。不同行业对数据中心的需求各有侧重：

- **互联网**：自动化程度较高，服务器大规模协同，高性能、高吞吐、高并发，自研管理/调度平台。
- **金融**：核心生产多自建并双活部署，非核心业务倾向于云化部署。
- **电信**：分布广体量小，SaaS客户群体的主力军。
- 其他行业：政府、制造、媒体、能源、零售、交通、教育、超算等。

#### 1.4 数据中心的演进趋势

**趋势一：从单中心到多地多中心**



数据中心的部署架构经历了四个阶段：

1. **单中心**：业务单中心运行。
2. **主备中心**：业务单中心运行，异地灾备（跨城市 > 200 KM）。
3. **两地三中心**：业务同城双活运行，异地灾备（同城 40 KM ~ 200 KM）。
4. **多地多中心**：核心系统主备，其他业务多中心多活。结合私有云与公有云。

**趋势二：从自用自维到出售云服务**

数据中心的IT架构也经历了四个阶段：

1. **传统DC**：三层网络架构，物理机，手工部署。
2. **虚拟化**：二层网络，虚拟机。计算虚拟化驱动网络虚拟化，SDN（软件定义网络）兴起。
3. **云化**：容器化使管理网元数量进一步扩张，云平台整合资源实现统一管理。
4. **资源服务化**：多样资源（容器、虚拟机等），强调对外开放，提供灵活可整合的服务能力，以及可计费的销售单元。分为 IaaS、PaaS、SaaS 三层服务模式。

**趋势三：多云部署**

根据 Flexera 2021 报告，92% 的企业明确多云战略，82% 的企业规划部署混合云。仅 7% 的企业使用单公有云/私有云。

**趋势四：算力密度和强度持续升级**

华为 GIV 预测 2030 年将迎来 YB 数据时代，对比 2020 年，通用算力增长 10 倍、人工智能算力增长 500 倍。多样性计算、绿色集约、智能认知、内生安全、多维协同是未来数据中心的五大趋势。

### 2. 数据中心网络概述

#### 2.1 数据中心设备

数据中心设备简单分为三类（不包括基础设施）：

| 类别 | 设备 |
| :-- | :-- |
| 计算设备 | 机架式服务器、刀片式服务器、高性能/人工智能计算集群 |
| 存储设备 | 集中式存储（磁盘阵列，全闪存/混合闪存）、分布式存储（块/文件/对象存储） |
| 网络设备 | 路由器、交换机、防火墙、负载均衡、IDS/IPS |

#### 2.2 典型数据中心组网



一个典型的数据中心组网包括：

- **DCN**（Data Center Network，数据中心网络）：IP 网络，负责数据中心内部互联及外部出口互联。
- **SAN**（Storage Area Network，存储区域网络）：FC/IP/IB 存储网络。
- **服务器**：计算节点和存储节点。

#### 2.3 数据中心网络架构演进



数据中心网络架构经历了三个版本的演进：

**DCN 1.0：STP + VRRP**

在 DCN 发展初期，采用经典的 STP + VRRP 方案：
- 汇聚交换机和接入交换机组建二层网络，使用 STP 破除二层环路。
- 网关部署在汇聚交换机上，采用 VRRP 提供冗余。
- **问题**：STP 依赖阻断端口破环，导致链路利用率低。

**DCN 2.0：堆叠 / M-LAG**

采用堆叠和 M-LAG 技术解决 STP + VRRP 的问题：
- 汇聚层：M-LAG 或者堆叠，同时作为 L3 网关。汇聚层与接入层间采用跨设备链路捆绑互联。
- 接入层：M-LAG 或者堆叠，满足服务器网卡绑定方式双归到两台物理设备的需求。
- **优势**：支持链路的完全负载分担。
- **问题**：规模受限，东西向 L3 流量非最优。

**DCN 3.0：Spine-Leaf + VXLAN**

采用 Spine-Leaf 组网架构结合 VXLAN 和 EVPN 技术：
- VXLAN 通过 MAC in UDP 的报文实现数据面封装，构建基于 IP 的 Overlay 虚拟局域网。
- Spine 和 Leaf 之间全互联，支持 4 台甚至更多的 Spine。
- **优势**：支持链路的完全负载分担，实现东西向 L3 流量最优路径。

#### 2.4 Spine-Leaf 架构



Spine-Leaf 是数据中心的新型网络架构，分为 Spine 节点和 Leaf 节点：

| 术语 | 解释 |
| :--: | :-- |
| Spine | 骨干节点，VXLAN Fabric 网络核心节点，提供高速 IP 转发功能，通过高速接口连接各个功能 Leaf 节点。 |
| Leaf | 叶子节点，VXLAN Fabric 网络功能接入节点，提供各种网络设备接入 VXLAN 网络的功能。 |
| Fabric | 一组 Spine 及 Leaf 节点互联组成数据中心基础物理网络拓扑。 |
| Service Leaf | 提供 Firewall 和 Load Balancer 等 L4~L7 增值服务接入 VXLAN Fabric 网络的功能，简称"SL"。 |
| Server Leaf | 提供虚拟化服务器、非虚拟化服务器等计算资源接入 VXLAN Fabric 网络的功能，简称"Leaf"。 |
| Border Leaf | 提供数据中心外部流量接入 VXLAN Fabric 网络的功能，用于连接路由器或者传输设备，简称"BL"。 |

**Spine-Leaf 的优势：**
- Leaf 与 Spine 实现三层全连接、等价多路径，提高网络的可用性。
- Spine-Leaf 架构具备极高的扩展能力，支持**多节点扩展**和**多级扩展**。

#### 2.5 同城双 DC 与多地多 DC

**同城双 DC — Multi-PoD 方案**



业务规模的增长使得单个数据中心的资源很难满足需求。同城双数据中心分为主备容灾和双活两种模式。Multi-PoD 方案可用于解决跨数据中心互通的自动化部署和跨数据中心的业务容灾多活的问题。

Multi-PoD 主要有两种部署场景：
1. 多个 PoD 在同一个 DC 内统一纳管，形成一个大资源池。
2. 同城近距两个 DC 形成一个大资源池，满足网络容灾的诉求。

**多地多 DC — Multi-Site 方案**



随着业务持续增长，企业数据中心规模从单数据中心到同城双中心、两地三中心甚至多地多中心演进。Multi-Site 是两个或者多个位于不同地域的 Site。每个 Site 都是一个独立的大二层网络域。

#### 2.6 Underlay 和 Overlay

**Overlay 网络：**
- 通过 VXLAN 在 Underlay 网络基础上建立的逻辑网络。
- 具有独立的转发平面和控制协议。
- 对于连接在 VXLAN 隧道端点之外的设备来说，Underlay 物理网络是透明的。

**Underlay 网络：**
- 由各类物理网络设备构成，属于 Overlay 网络的承载网。
- 为上层 Overlay 网络提供可达性、可靠性等基础能力。
- 控制平面协议一般采用 OSPF 或 eBGP，转发平面协议一般为 IPv4。
- Underlay 与 Overlay 逻辑隔离，其并不感知 Overlay 网络路由。

### 3. 数据中心基础设施

#### 3.1 综合布线 ToR / EoR / MoR



数据中心综合布线有三个重要概念：

- **ToR**（Top of Rack）：交换机安装在机架顶部，服务器在本机架内完成接入。布线距离短，管理方便。
- **EoR**（End of Row）：交换机安装在列头，服务器通过列头交换机完成接入。适合服务器密度较高的场景。
- **MoR**（Middle of Row）：交换机安装在机列中部，是 ToR 和 EoR 的折中方案。

#### 3.2 机房模块

以某金融数据中心为例，每个建筑每层划分成多个机房模块：

- **网络模块**：主要负责网络接入，作为广域网和局域网的核心。
- **存储模块**：主要用来集中放置存储设备。
- **开放服务器模块**：主要用来集中放置服务器。
- **测试模块**：主要用来集中放置测试设备。

每层按照供电密度划分为高密度、中密度、低密度三个区域。

#### 3.3 PoD



**PoD**（Point of Delivery）是数据中心常用的物理设计概念，是网络、存储、计算一体模块化设计的实体。为了便于数据中心的资源池化操作和管理，可将一个数据中心划分为一或多个物理分区。

可以根据实际业务需要定义 PoD：
- 大型数据中心可以以整个机房模块为单位定义 PoD。
- 中型数据中心可以两排或多排机柜为单位定义 PoD。
- 小型数据中心以一个或数个机柜灵活组成一个 PoD。

#### 3.4 数据中心交换机



数据中心交换机一般指硬件交换机，发展经历了三个阶段：

| 阶段 | 时间 | 接入方式 | 上行带宽 | 说明 |
| :--: | :--: | :--: | :--: | :-- |
| 阶段一 | 约2000~2006年 | FE/GE | GE | 数据中心服务器以 FE/GE 接入为主，满足小型数据中心 |
| 阶段二 | 约2006~2012年 | GE/10GE | 10/40GE | 数据中心服务器以 GE/10GE 接入为主，满足中、小型数据中心 |
| 阶段三 | 约2012~2020年 | 10/40GE | 100GE | 数据中心服务器以 10/40GE 接入为主，核心交换机提供高密度接口 |

#### 3.5 面向 AI 时代的交换机

CloudEngine（CE）系列是华为公司面向下一代数据中心的高性能交换机：

- **框式交换机**：CE 16800 和 CE 12800 系列，主要用于数据中心核心高速转发。
- **盒式交换机**：CE 9800/8800/6800/5800 系列，主要用于数据中心高密度接入。

面向 AI 时代的数据中心网络除了部署交换机外，推荐使用**网络控制器**（iMaster NCE-Fabric）和**分析器**（FabricInsight）实现智能部署与运维。

### 4. 计算与虚拟化

#### 4.1 服务器虚拟化

服务器虚拟化（Server Virtualization）通过虚拟化技术在一台物理服务器上运行多台虚拟机（Virtual Machine），以获得物理资源利用率提高、业务快速部署和弹性等优势。用户可以在虚拟机上安装和运行各种应用和服务，就像使用物理服务器一样。

#### 4.2 虚拟化管理平台

随着业务增长，虚拟化服务器集群规模达到几百数千台，需要使用虚拟化管理平台进行统一管理。虚拟化管理平台提供：
- 简单的用户界面
- 监控和管理虚拟化资源
- 简化虚拟机的创建流程
- 配置资源调度策略和执行规则

业界的虚拟化平台有华为 VRM、VMware vCenter、微软的 System Center 等。

#### 4.3 云计算与 OpenStack

OpenStack 是开源的云操作系统，可控制整个数据中心的大型计算、存储和网络资源池。部署 OpenStack 后，用户能够通过 Web 界面、命令行或 API 接口管理资源。

OpenStack 不是云计算，而是构建云计算的关键组件——云平台。其目标是资源管理，支持管理异构厂家的计算、存储和网络资源池。

**OpenStack 核心组件：**

| 组件 | 功能 |
| :--: | :-- |
| Nova | 管理计算资源 |
| Cinder | 管理块存储资源 |
| Neutron | 管理网络资源 |

云平台上层接入云管理平台，功能包括：面向租户的运营、发放云服务、计费、多云监控等。

#### 4.4 容器



**容器**（Container）是一种操作系统级别的虚拟化技术。与虚拟机相比，容器更轻量和高效。容器的好处在于可以打包应用及其运行环境，使得"开发-测试-部署-运维"的过程极大简化。

**容器与虚拟机对比：**

| 对比项 | 虚拟机 | 容器 |
| :-- | :-- | :-- |
| 隔离级别 | 完整的 OS 级隔离 | 进程级隔离 |
| 内核 | 独立 Guest OS | 共享 Host OS 内核 |
| 启动速度 | 分钟级 | 秒级 |
| 资源占用 | 较高（每个 VM 需独立 OS） | 较轻量 |
| 独立性 | 强 | 较弱 |

**容器管理平台：**

企业数据中心或公有云管理的节点规模庞大，需要独立的容器管理平台。一个成熟的容器管理平台至少包括应用编排管理和集群资源调度两个功能。业界主要有三种平台：Kubernetes、Swarm、Mesos。

### 5. 存储与 AI 计算

#### 5.1 存储类型



在企业数据中心存储中经常提及块存储、文件存储和对象存储三个概念：

| 存储类型 | 典型应用 | 典型设备形态 | 运用场景 |
| :-- | :-- | :-- | :-- |
| 块存储 | 要求高性能的应用 | 磁盘阵列，硬盘 | 需要高 IO 的数据库 |
| 文件存储 | 需局域网共享文件的应用 | FTP、NFS 服务器 | 普通场景：文件共享；高性能场景：视频处理、动画渲染、高性能计算 |
| 对象存储 | 数据量大、存储容量增速快的应用 | 内置大容量硬盘的分布式服务器 | 点播/视频监控、图片存储、网盘、静态网页、异地备份/归档等 |

**存储产品分类：**

- **集中式存储**：由一台或多台主计算机组成中心节点，数据集中存储。根据介质不同可分为全闪存和混合闪存存储。
- **分布式存储**：将数据分散存储在多台独立设备上。采用可扩展的系统结构，利用多台存储服务器分担存储负荷，提高了系统的可靠性、可用性和存取效率。

#### 5.2 高性能计算（HPC）

HPC（High Performance Computing）是计算机科学的一个分支，通过集群架构、并行算法和相关软件以并行计算/分布式计算的方式实现单台计算机无法达到的运算速度（每秒万亿次以上）。

典型的 HPC 架构包括：基础设施、计算节点、存储及文件系统、网络交换、集群管理、资源调度等。

**算力衡量单位**：FLOPS（Floating-point Operation Per Second，每秒所执行的浮点运算次数）。

| 单位 | 含义 |
| :--: | :-- |
| 1 K FLOPS | 10^3，每秒一千次 |
| 1 M FLOPS | 10^6，每秒一百万次 |
| 1 G FLOPS | 10^9，每秒十亿次 |
| 1 T FLOPS | 10^12，每秒一万亿次 |
| 1 P FLOPS | 10^15，每秒一千万亿次 |
| 1 E FLOPS | 10^18，每秒一百京次 |

#### 5.3 AI 计算对网络的需求



机器学习和深度学习伴随着强大计算需求，通常会组建分布式计算集群。当前主流使用 100 Gbps 及以上的网络设备搭建。AI 性能测试结果表示，网络会极大影响计算性能。

**AI 计算对网络的核心需求：**
- 高带宽、低时延、无丢包
- Incast 场景下流量控制
- 快速响应的拥塞控制
- 快速有效的流负载均衡机制
- 混合流量中大小流的区分调度

### 6. 数据中心关键网络技术



#### 6.1 负载均衡（SLB）

负载均衡（Load Balancing）是一种用于计算机集群分配负载的技术。它使用代理设备接收外部请求，然后将该请求分担到内部多台服务器。

- **三层负载均衡**：基于 IP 实现
- **四层负载均衡**：基于 IP + 端口实现
- **七层负载均衡**：基于应用层协议（如 HTTP）实现

在数据中心中的应用：
- **SLB**（Server Load Balancing）：实现数据中心内的服务器负载分担。
- **GSLB**（Global Server Load Balancing）：实现数据中心间的负载分担。

#### 6.2 M-LAG

M-LAG（Multichassis Link Aggregation Group，跨设备链路聚合组）是一种实现跨设备链路聚合的技术：

- 把链路可靠性从单板级提高到设备级
- 一方面起到负载分担流量的作用
- 另一方面起到备份保护的作用

在数据中心中一般由两台 ToR 协商为主备组建 M-LAG，负责其他设备（服务器或防火墙）接入。在 Spine-Leaf 架构中，M-LAG 被推荐用于服务器或防火墙接入，满足高可靠性需求。

#### 6.3 VXLAN

VXLAN 本质上属于一种 VPN 技术，能够在任意路由可达的网络上叠加二层虚拟网络：

- VXLAN 底层依赖的路由网络无网络结构限制，具有扩展能力
- VXLAN 报文中存在 **VNI**（VXLAN Network Identifier）字段，类似于 VLAN ID，用于区分不同网络
- 两台设备之间一般只有一条 VXLAN 隧道，类似于 Trunk 链路

**VXLAN 在云数据中心的应用：** 虚拟机能够在服务器集群内任意地热迁移。当虚拟机迁移到另一个 Leaf 节点下的服务器时，虚拟机的 IP、MAC 保持不变，从而实现业务访问不中断。Spine-Leaf + VXLAN 是此场景下的最佳实践。

#### 6.4 EVPN

EVPN 最初在 RFC 7432 中定义，作为 VXLAN 的控制平面：

- **MPLS Based EVPN**：控制面 EVPN MP-BGP，数据面 Label Switching（MPLS）或 IP/GRE tunnel
- **NVO（Network Virtualization Overlay）**：控制面 EVPN MP-BGP，数据面 VXLAN

在数据中心使用 NVO 解决方案，即 BGP EVPN 作为控制平面传递 DC 内和 DC 间路由，VXLAN 作为数据转发平面转发数据报文。EVPN 支持 DC 内和 DC 间 L2 或者 L3 流量互通。

#### 6.5 Telemetry

Telemetry（网络遥测技术）是一项远程的从物理设备或虚拟设备上高速采集数据的技术：

- 与 SNMP 相比，Telemetry 采集周期达到**亚秒级**
- 设备通过**推模式**（Push Mode）周期性地主动上送信息
- 提供了更实时更高速的数据采集功能

**在数据中心的应用：** 基于 Telemetry 采集设备高精度数据，构建智能运维系统。包括网络业务分析（性能分析、容量分析、故障分析、安全分析、路由分析）和可视化工具集成管理。

#### 6.6 NETCONF

NETCONF（Network Configuration Protocol，网络配置协议）提供一套管理网络设备的机制。用户可以使用这套机制增加、修改、删除网络设备的配置，获取网络设备的配置和状态信息。

**NETCONF vs SNMP vs CLI：**

| 功能 | NETCONF | SNMP | CLI |
| :-- | :-- | :-- | :-- |
| 接口类型 | 机机接口（标准规范） | 机机接口 | 人机接口 |
| 操作效率 | 高（对象建模，一次交互） | 中 | 低 |
| 扩展能力 | 可扩展协议私有能力 | 弱 | 一般 |
| 事务处理 | 支持试运行、出错回滚 | 不支持 | 部分支持 |
| 安全传输 | SSH、TLS 等多种协议 | 仅 SNMPv3 | 支持 SSH |

在数据中心中 NETCONF 主要被用于网络控制器进行业务编排后向南向设备下发配置，消息使用 XML 编码。

#### 6.7 微分段

微分段（Microsegmentation）是一种精细分组的安全隔离技术：

- 通过将数据中心业务单元按照一定规则进行分组，在分组间部署策略来实现流量控制
- 传统数据中心基于 VLAN/VNI 进行子网划分，粒度较粗
- 微分段可以提供更细粒度、更灵活的分组方式，例如基于 IP、MAC、虚拟机名等

在数据中心中，微分段用于将不同的服务器或 VM 分组，然后制定各分组之间的访问策略，实现不同业务节点之间的流量控制。既可以通过 CE 交换机本身实现，也可以通过 CE 交换机与 iMaster NCE-Fabric 控制器配合使用。

#### 6.8 业务链

业务链（Service Function Chain，SFC）是一种给应用层提供有序服务的技术：

- 通过定义业务链路径，使匹配的流量依次经过指定的 VAS 设备（如防火墙、负载均衡、深度检测、入侵防御等）
- 依次获取相应的增值服务
- 业务链在控制器上编排，可以通过 PBR 或者 NSH 实现

使用业务链有助于使用同一个网络，对不同的业务提供差异化的增值服务。

#### 6.9 智能无损网络

智能无损网络通过 AI Ready 的硬件架构及智能无损算法，在保证网络零丢包的基础上实现最高吞吐和最低时延。

智能无损网络技术架构包含五层：
1. **硬件层**
2. **流控层**：PFC、PFC storm control、PFC deadlock free
3. **拥塞控制层**
4. **流量调度层**
5. **应用加速层**

在数据中心中可用于不同计算场景：HPC 高性能计算网络、AI 分布式训练网络，一般部署为封闭网络。业务网络通常部署为 TCP/IP 有损网络。

## 结语

数据中心的技术演进，看似庞杂，其实方向始终清晰：用简单的方法，解决复杂的问题。从传统三层架构到 Spine-Leaf + VXLAN，从单中心到多地多中心，从手工部署到自动化运维，每一次演进都在追求更高的效率和更好的体验。技术永远在变，但思考的框架比知识点更重要。希望这篇内容能帮你抓住主线，在快速迭代的领域里找到自己的节奏。感谢阅读，一起成长。
